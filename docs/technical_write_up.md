## 1. Project Overview

**Goal:** Build a chatbot that accepts a cybersecurity question, sends it to a free LLM via OpenRouter, receives the response, and displays it in a clean interface with basic conversation memory.

**Stack:** Python, OpenRouter API, (later) Streamlit or Django.

**Learning arc:** plain request/response → memory → prompt engineering → web UI → refactor.

## 2. Core Concepts Learned

### 2.1 How an LLM Chatbot Works (9-step round trip)
1. User types a message in the UI.
2. Frontend sends message to backend.
3. Backend formats message into JSON.
4. Request goes over HTTPS to LLM provider.
5. Provider runs the model on GPUs.
6. Model generates response token by token.
7. Response is sent back to the backend.
8. Backend returns it to the frontend.
9. UI displays it.

**Takeaway:** The Python backend sits in the middle. Swap the UI, the middle stays the same. That is the heart of this project.

### 2.2 What is OpenRouter?
- A **unified API gateway** for many LLMs.
- One account, one API key, dozens of models.
- **OpenAI-compatible API format** → any OpenAI client code works.
- Free tier exists (rate-limited). Paid = per-token.
- Mental model: *hotel concierge for LLMs* — you pick the room, OpenRouter handles the rest.

### 2.3 Essential Vocabulary

| Term | Meaning |
|------|---------|
| **API Key** | Secret authentication string. Treat like a password. Stored in `.env`, never in code. |
| **Model** | The specific LLM you talk to (Qwen, Llama, Gemma…). Switched by changing a string. |
| **Tokens** | Chunks of text the model reads/writes (~0.75 words each in English). |
| **Context Window** | Max input + output combined. 8K / 32K / 128K / 262K. |
| **System Prompt** | A `role: system` message that sets the AI's behavior, tone, and limits. |
| **Chat Messages** | A list of `{"role": ..., "content": ...}` objects: `system` / `user` / `assistant`. |
| **Temperature** | Randomness dial. Low (0.2) = focused, High (1.0) = creative. ~0.3 for security. |
| **max_tokens** | Cap on response length. |
| **stream** | Receive tokens as they are generated (used for "typing" effect). |

### 2.4 Request Shape (what we will send)
```json
{
  "model": "google/gemma-4-31b",
  "messages": [
    {"role": "system", "content": "You are a cybersecurity expert..."},
    {"role": "user", "content": "What is XSS?"}
  ],
  "temperature": 0.3
}
```
Response carries the assistant's text in `choices[0].message.content`.

## 4. Why This Project Is a Good First Step

- Chat is the **simplest LLM pattern**: one input, one output. Everything else is a variation.
- **RAG systems** = same architecture, but the user's question first retrieves documents that get injected into the message list.
- **AI Agents** = the model decides to call a tool → tool returns data → model reasons → repeat. Same backend, smarter loop.
- Mastering this round-trip = the foundation for applied AI work.

---

## 5. Common Beginner Mistakes to Avoid

1. **Hardcoding the API key** in source files → leaks to Git, gets abused. Use `.env`.
2. **No error handling** → free models rate-limit, networks fail, providers go down. Always wrap calls in `try/except`.
3. **Sending full conversation every time without trimming** → blows past context window, costs more.
4. **Vague system prompts** ("be helpful") → weak, inconsistent outputs. Engineer them properly.
5. **Treating the model as a database** → LLMs hallucinate. Real security intel eventually needs RAG.

## 6. Decisions Log

### Decision 1 — Model choice
- **Date:** 2026-06-12
- **Options considered:**
  - `google/gemma-4-31b` (free) — 262K context, strong reasoning, function calling
  - `qwen/qwen3-next-80b-a3b-instruct:free` — 262K context, fast MoE
  - `meta-llama/llama-3.3-70b-instruct:free` — 131K, battle-tested
  - Venice Uncensored — rejected: weak safety, wrong fit for a defensive learning tool
  - Poolside Laguna M.1 — rejected: coding-specialist, overkill for Q&A
- **Chosen:** `google/gemma-4-31b` as primary.
- **Backup:** `meta-llama/llama-3.3-70b-instruct:free` if primary is rate-limited.
- **Reasoning:** 262K context, strong reasoning, Google-grade quality, useful function-calling for future RAG/agent phases.

### Decision 2 — API key handling
- **Date:** 2026-06-12
- **Problem:** Initial key was pasted into chat → considered compromised.
- **Action taken:** Key rotated in OpenRouter dashboard before any code was written. New key stored only in `.env`.
- **Rule going forward:** Key lives in `.env` only. `.env` is added to `.gitignore`. Code reads it via `python-dotenv`. Never paste the real key in chat, screenshots, or source files.

### Decision 3 — Project layout (Phase 2)
- **Date:** 2026-06-12
- **Chosen:** Engine/interface split. Reusable logic in `app/`. Two separate frontends (`phase3_cli/`, `phase6_web/`) that both import from `app/`. Tests in `tests/`. Study logs in `docs/`.
- **Trade-off accepted:** The phase-named frontend folders are a teaching scaffold. They will be collapsed into a cleaner final structure in Phase 7.
- **Reasoning:** Mirrors the Django pattern of separating services from views. Keeps the CLI and the Streamlit app from duplicating the OpenRouter call.
- **Settled in Phase 7 (2026-06-12):** `phase3_cli/` → `cli/`, `phase6_web/` → `web/`. The teaching scaffold has served its purpose. See Decision 7.

### Decision 4 — Test framework
- **Date:** 2026-06-12
- **Chosen:** `unittest` (stdlib) for now, no extra dependency.
- **Reasoning:** Zero install cost, ships with Python, sufficient for the smoke tests we need in Phase 2. We can switch to `pytest` later if test suites grow.

### Decision 6 — Three-layer split (engine → UI-logic → view) (2026-06-12)

**Chosen:** the Streamlit view (`web/streamlit_app.py`) is a thin rendering layer. All "how do I turn app state into a request / a transcript / a bubble" logic lives in pure functions in `web/chat_helpers.py`. The engine (`app/`) is unchanged. (Folder name corrected from `phase6_web/` → `web/` in Phase 7; no logic changes.)

**Why split at all?**
- Streamlit reruns the whole script on every interaction. Any logic that lives next to `st.markdown(...)` is hostile to test — you cannot unit-test a Streamlit widget without spinning up a browser. Moving the logic out of the view lets the test suite import it like a normal Python module and assert on real values.
- The helpers are **pure**: no `streamlit` import, no I/O, no globals. That keeps the test runtime fast (the whole `ChatHelpersTests` class runs in milliseconds) and keeps the helpers' behaviour obvious — you can read 10 lines and know exactly what the function does.
- Keeping the view thin keeps the *engine contract* clean. The view never reaches into `requests` or constructs an OpenRouter payload directly. The only function it calls is `app.openrouter.chat(messages, ...)`. When Phase 7 refactors, neither the helpers nor the engine have to change.

**The three layers, top to bottom:**
1. `web/streamlit_app.py` — rendering, state, sidebar controls, chat input, error banner, status line.
2. `web/chat_helpers.py` — `_build_messages`, `_truncate_history`, `_serialize_for_download`, `_count_chars`, `_bubble_alignment`. All pure. All tested.
3. `app/` — `openrouter.chat`, `config`, `prompts`. The engine, unchanged since Phase 5.

**Verification:**
- `tests/test_smoke.ChatHelpersTests` — 10 tests covering the happy path, the truncation invariants, the transcript format, char counting, and bubble alignment. 27/27 green.
- `streamlit run web/streamlit_app.py` boots cleanly (`You can now view your Streamlit app…`); byte-compile of both files is `SYNTAX OK`.

### Decision 5 — Phase 5 system prompt scope (2026-06-12)

**Chosen:** a four-pillar teaching prompt (`SecTutor`) covering defensive security, DevSecOps, AI/ML security, and offensive-security education. The prompt explicitly refuses to generate working exploit code, malware, droppers, or step-by-step compromise instructions for specific real systems.

**Why this scope:**
- A defensive-only prompt would be a poor fit for a *cybersecurity* student. Knowing the structure of classic attacks is the foundation of being able to defend against them in code review and design.
- A "do anything" prompt with no guardrails would not be a teaching tool — it would be a weaponized tool. The boundary is on purpose, and it is recorded here so future readers (and the user) can see the reasoning, not just the rule.

**What the prompt covers (the four pillars):**
1. **Defensive security** — threat modeling, IR, hardening, IAM, network security, applied cryptography.
2. **DevSecOps** — secure SDLC, SAST/DAST, dependency and supply-chain security, secrets management, container/Kubernetes hardening, IaC scanning.
3. **AI / ML security** — prompt injection, model exfiltration, training-data poisoning, OWASP LLM Top 10, AI red-teaming fundamentals, agent safety.
4. **Offensive-security education** — terminology, attack classes (injection, XSS, CSRF, SSRF, deserialization, race conditions, privilege escalation), the *structure* of an attack, *why* it works, and *what defeats it*. Concept-level, not turn-key-weapon-level.

**Where the line is drawn (also encoded in `app/prompts.py`):**
- *Allowed:* explaining the structure of an attack class, citing OWASP/MITRE ATT&CK/NIST references, illustrative labeled snippets, defensive countermeasures, recommended authorized lab environments (HackTheBox, TryHackMe, PortSwigger Academy, WebGoat, DVWA).
- *Refused:* working exploit code, malware, ransomware, droppers, C2 tooling, payloads targeting a specific real WAF/EDR/MFA, step-by-step instructions to compromise a real system the user does not own or have permission to test.

**Verification (live probes on 2026-06-12 against `google/gemma-4-31b-it:free`):**

### Decision 6 — Two-profile system prompt (2026-06-13)

**Chosen:** keep the defensive four-pillar prompt as the **default** system message, and add a second prompt (`OFFENSIVE_MENTOR_SYSTEM_PROMPT`, persona "SecMentor") that the user can switch to from a sidebar **Teaching mode** radio. Unknown / missing / non-string `teaching_mode` values always fall back to the defensive prompt — the helper fails closed.

**Why a second profile, not just a softer guardrail on the default:**
- A defensive-only prompt makes the chatbot a poor fit for a learner *stuck* on TryHackMe or HackTheBox. The user's question on those platforms is often "give me the SQLi payload form for the login form on 'Lame'", and a "no working code ever" rule blocks exactly that question. The point of the chatbot is to teach the structure, including the runnable shape, in a controlled scope.
- Loosening the *default* prompt would have been the wrong move: most users want the safer, concept-level assistant, and the few who want the wider authorization should have to opt in by an explicit, visible UI control (not by finding a magic phrase in the prompt).
- A second prompt, gated by a sidebar radio, gives the user (a) a clear "you are in the wider mode" cue, (b) a one-click way back to the safer default, and (c) a place to document the boundary in-app via the existing "About / scope" expander.

### Phase 4 — Concept: history is just a list

**Mental model.** An LLM has no memory between calls. Every "memory" you see in a chat product is the application *re-sending prior turns* inside the `messages` list on each new call. There is no hidden state on the model side. This is also why the same prompt can produce different answers if you re-send it — randomness is per-call, not persistent.

**What we changed in `cli/chatbot.py`:** (folder name corrected from `phase3_cli/` → `cli/` in Phase 7)
- A `history: list[dict]` accumulator starts with the system prompt and grows one user/assistant pair per turn.
- Each call: `messages = history + [{"role": "user", "content": new_input}]` is sent, then the assistant's reply is appended to `history`.
- `/clear` resets history to just the system prompt without restarting the process.
- A running `[history: N messages, ~M chars]` line is printed after each reply so the user can watch the context grow.

**Why this matters beyond this project:**
- **RAG** = inject retrieved documents into the same `messages` list (typically as a `user` turn that says "context: ..." before the real question).
- **Agents** = the model emits a tool call, the tool result becomes a new turn in the same list, the loop continues.
- **Token cost** grows linearly with history. With free models the limit is rate limits; with paid models it is money. We did not cap history in Phase 4 because Gemma 4 31B IT has a 262K context window — there is room to grow — but the `MAX_HISTORY_MESSAGES` knob is the next thing to add when we want to talk to a smaller-context model.

**Why no engine changes were needed.** `app/openrouter.py` is stateless — it just sends a `messages` list. The CLI owns the list. This is the engine/interface split paying off: the same `chat()` call works for a one-shot question or a 50-turn conversation, because the engine does not care.

## 9. Glossary (build as we go)

- **LLM** — Large Language Model. A neural network trained on text that predicts the next token.
- **Inference** — Running a trained model to produce output (as opposed to training it).
- **Endpoint** — A URL an API exposes for a specific operation (e.g., `https://openrouter.ai/api/v1/chat/completions`).
- **JSON** — The data format we send to and receive from the API.
- **`.env` file** — A plain-text file holding key=value config, loaded by `python-dotenv`.
- **`.gitignore`** — Tells Git which files to never track (secrets, virtual envs, caches).
- **Two-pass pattern** — In Streamlit, splitting a "set up state, run a slow call, render the result" flow into two script runs: pass 1 mutates `session_state` and calls `st.rerun()`; pass 2 reads the state and renders. The state machine is `pending_request` (and any other transient flags) in `session_state`.
- **Cold-start model** — The model used on the first run of the chat, taken from `OPENROUTER_MODEL` in `.env`. Distinct from the sidebar selector, which controls the model on the next click.
- **Friendly error** — A user-facing error message produced by classifying the raw exception at the helper layer (`build_user_turn_text` / `safe_call_llm`) so the view never has to render a stack trace. Decoupled from the OpenRouter error taxonomy so the chat view stays declarative.
- **RAG (Retrieval-Augmented Generation)** — A pattern where, before each LLM call, the system retrieves relevant chunks from a vector store and injects them into the prompt as a synthetic prior `user` turn (`Context: <chunk 1>\n---\n<chunk 2>\n...`). The model answers the question *with the retrieved context in its working window* rather than from its training data alone. Phase 12.
- **FAISS** — Facebook AI Similarity Search. A library for fast nearest-neighbor search in high-dimensional vector spaces. In this project: `faiss.IndexFlatIP` over L2-normalized 384-dim embeddings, used for inner-product (= cosine similarity) search. Per-chat index for uploaded files (`app/rag_store.py`), separate global index for the security corpus (`app/rag_global.py`).
- **Chunk** — A contiguous slice of text, ~800 chars with ~120 char overlap (`app/rag_chunker.py:chunk`), produced by `app/rag_chunker.py:chunk(text, chars_per_chunk=800, overlap=120)` with word-boundary respect so a chunk does not split a word in half. Each chunk becomes one row in `chat_chunks` / `global_chunks` and one vector in the FAISS index.
- **Embedding** — A fixed-length dense float vector (here, 384 dims, `EMBEDDING_DIM=384` in `app/rag_embedder.py`) produced by a sentence-transformers model (`all-MiniLM-L6-v2` by default). The vector represents the *semantic* content of the source text — text with similar meaning has vectors close in cosine similarity.
- **L2 normalization** — Scaling a vector so its L2 norm is 1. Required before using `faiss.IndexFlatIP` for cosine similarity (`app/rag_embedder.py:l2_normalize`). Inner product on unit vectors equals cosine similarity.
- **Score threshold** — The minimum cosine similarity (`DEFAULT_SCORE_THRESHOLD = 0.30` in `app/rag_embedder.py`) above which a retrieved chunk is considered "more similar than not" and passed into the context-injection step. Below the threshold, the chunk is treated as noise and dropped. Chosen empirically against a 5-query smoke corpus.
- **Degraded mode** — When `PUKU_RAG_OFFLINE=1` is set, `app/rag_embedder.py` returns a 384-dim zero vector and the store short-circuits `search()` to an identity search (returns the most recently added chunks in insertion order, score 0.0). Lets the test suite run on CI without a sentence-transformers model download and lets a developer keep working when the model registry is unreachable.
- **Version counter (`_index_version`)** — An integer on the `chats` table, incremented on every `add_chunks_returning_ids` / chunk-rebuild. `RagStore(chat_id)` caches the FAISS index keyed by the counter, and a stale cache is detected by comparing the counter at `search()` time to the counter at `add()` time. No manual `rebuild()` call needed — invalidation is automatic.
- **Cross-chat isolation** — The invariant that a chunk added to chat A never appears in chat B's search results, even when the query vector is identical. Enforced by `RagStore(chat_id)` being constructed per chat and by `tests/test_rag.py:CrossChatIsolationTests` (3 tests, pinned) as the regression fence.
- **Global security corpus** — A second parallel FAISS index (`app/rag_global.py:GlobalCorpusStore`) holding curated reference material from five source kinds: OWASP (Top 10 + Cheat Sheet series), MITRE ATT&CK (techniques + mitigations), CWE (top 25 + targeted weakness entries), GTFOBins (Unix binaries for privilege escalation), Sigma (detection rules). Built once by `scripts/ingest_security_corpus.py`, deduped by SHA-256, queried alongside the per-chat index on every turn.
- **Source citation** — A row in the `sources` table `(message_id, chunk_id, source_kind, source_label, score)` recording which chunk backed which assistant turn. The view renders the citations as a `Sources: chat upload (1) + global corpus (2)` footer in the assistant bubble so the user can audit which context the model saw.
- **Source filter** — A `source_filter` parameter on `GlobalCorpusStore.search(query, k, source_filter)` that restricts the global corpus to one or more source kinds. A user asking "what does OWASP say about SQL injection?" can pin the answer to OWASP only.
- **Scored-then-merge** — The retrieval merge order in `web/chat_helpers.py:build_messages_with_rag`: per-chat top-k, then global corpus top-k, each returning its own similarity scores, then the helper picks the top-N across both (default 3 from per-chat + 2 from global, total 5). Not concatenated — the two score spaces are interleaved by similarity.
- **`touch_chat(chat_id)`** — A `app/storage.py` call that bumps `updated_at` on a chat. Called on every successful send so an actively-used chat floats to the top of the sidebar's `updated_at DESC` ordering.
- **Two-step soft delete** — The trash pattern in the chat sidebar: a confirmation `st.warning` is required before the row is flipped to `soft_delete=1`; the soft-deleted row stays in the list (with a faded style) for one session, then is hard-deleted in the nightly cleanup. Hard delete cascades through `messages` / `artifacts` / `chat_chunks` / `sources`.
- **Smoke corpus** — A small hand-written fixture corpus (`smoke_corpus_inline.py` with 3 OWASP-like items, `smoke_corpus_debug.py` with 10 cross-source items) that lets a developer exercise `GlobalCorpusStore.search()` and `RagStore.search()` without the network fetch. Used by `tests/test_rag.py` and `tests/test_rag_global.py` for the cross-source / source-filter / clear-and-readd / degraded-mode tests.

### Decision 9 — Cross-cutting change-log file (`improvements.md`) + "always test the cache" pre-flight (2026-06-13)

**Chosen:** add a top-level `improvements.md` change-log alongside the per-phase narrative (`docs/technical_write_up.md`, `docs/my_first_ai_journey.md`), and ship the row-12 pre-flight script as a first-class artifact (`verify_changes.py`) rather than a one-off command sequence.

**Why a separate `improvements.md` and not just another row in §7:**
- §7's problems log is a *chronicle*. Each row is a single fix on a single date. It is the right shape for "what broke and what I did about it", but the wrong shape for "what is the current state of the offensive / defensive boundary, and how do I change it without breaking the contract."
- The user asked for a file that covers three orthogonal axes — (1) the offensive behaviour (the `OFFENSIVE_MENTOR_SYSTEM_PROMPT` persona, the swap plumbing, the 16 new tests, how to tune it), (2) the requirement-issue solutions (5 keys, multi-model, friendly errors, two-pass, pre-flight), and (3) future improvement suggestions. A single §7 row cannot hold that structure without becoming a 4-page cell that no one will scroll through.
- The cross-cutting shape (offensive ↔ defensive tuning knobs, ordered by blast radius; requirement solutions mapped to the file that owns them; future work as a prioritised list) is *re-readable* in a way a §7 row is not. It is the artifact the next session will reach for.

**Why `verify_changes.py` ships as a script, not as a §7 footnote:**
- Row 12's workflow rule has three checks (worker PID, port PID, fresh-import symbol). Re-typing those three PowerShell + Python one-liners at the start of every session is exactly the kind of manual ceremony that gets skipped. A 154-line script is the smallest possible automation that turns the rule into a single command (`python verify_changes.py`).
- The script has three useful properties that an inline command sequence does not: (a) it returns an exit code (0/1/2/3) so it slots into a future pre-commit hook or CI step; (b) it never echoes secret values — it only prints PIDs, port numbers, and importable symbol names; (c) its `READY` / `STALE_WORKER: kill PIDs …` / `IMPORT_BROKEN` verdicts are machine-parseable for a future dashboard.
- The script depends only on stdlib (`subprocess`, `sys`, `importlib`, `pathlib`) — no extra `requirements.txt` line. That keeps the install path unchanged.

**The "always test the cache" rule (the new, smaller decision embedded in Decision 9):**
- The row-12 bug was *a test that did not test the running app*. `python -m unittest discover` runs in a fresh process; it sees the latest source. The running Streamlit worker, however, holds the module object it imported *when it started*. The two diverged silently.
- The pre-flight script is the practical answer: *never start a long-lived server without first confirming a fresh interpreter can see the symbols it depends on*. The script turns "test the source" into "test the source *and* test the process state", and the verdict line tells you which one failed.
- Layered on the row-7 / row-10 / row-11 / row-12 rules: a green test suite proves the *code* is correct; a green pre-flight proves the *code + the process* is in sync. Both are required before a meaningful `streamlit run`.

**The durable lesson:** *if a workflow rule is invoked more than twice in the same project, encode it as a script, not as a comment*. A comment is read only by the person reading the file. A script is run by every future session, including the one that does not know the comment exists.

### Decision 10 — The content-shape seam is `build_user_turn_text`'s return type (2026-06-15)

**Trigger.** Row 14 shipped file uploads *as a feature*. Row 15 surfaced the next question: *"can it work like ChatGPT — the model can actually read the contents?"* The diagnostic walkthrough ended at one specific symbol: `build_user_turn_text` returns `str`. The OpenAI/chat-completions multimodal schema needs `content: [parts]`. The current return type is the ceiling for everything multimodal in this project.

**Chosen:** every future multimodal capability — text-file inlining, PDF text extraction, image base64-encoding, audio transcription, file persistence across turns — passes through *one* widened seam: `build_user_turn_text` (or its successor) returns `str | list[dict]`. The widening is backwards-compatible at the type level (existing string callers still get a string for text-only inputs) and forwards-compatible at the engine level (`_build_messages` learns to pass the list through to `app.openrouter.chat`, which learns to emit the OpenAI structured `content` schema). The router learns a `supports: "vision"` flag; the model pool gains at least one vision-capable entry. The user-facing UX (a `📎` button on `chat_input`) does not change.

**Why one seam, not one fix per file type:**
- A `for f in files: ...` loop in the *view* would scatter the file-shape logic across two layers (the file picker *and* the renderer), neither of which is unit-testable without a browser. Keeping the helper as the single fan-in point means the file-shape decision is testable as a pure function: `assert build_user_turn_text("explain", [png_file]) == [{"type": "text", "text": "explain"}, {"type": "image_url", "image_url": {"url": "data:image/png;base64,..."}}]`.
- The helper already owns the per-file decision (`_classify_upload`). The widening is the *promotion* of that decision from "do we inline the body as text?" to "do we emit a `text` part, an `image_url` part, or a placeholder `text` part?" — same logic, different output schema.
- The engine already accepts `messages: list[dict]`. The OpenRouter client already POSTs JSON. The only new wire concept is the *shape of `content`*. Pushing that shape through the helper — and *only* the helper — means `app/openrouter.py` gets exactly one new branch in `_build_payload`, not a rewrite.
- A new file format (`.eml`, `.ics`, `.pcap` — pick one) is a new `_classify_upload` case + a new part-type branch in the helper. The view, the engine, and the router do not change. That is what "seam" means here: the set of files that need to change for a new feature is small and well-defined.

**The widening, schematically:**
```
# today
def build_user_turn_text(text, files) -> str: ...

# after P3
def build_user_turn_content(text, files) -> str | list[dict]:
    parts: list[dict] = []
    if text: parts.append({"type": "text", "text": text})
    for f in files or []:
        if _classify_image(f):
            parts.append({"type": "image_url",
                          "image_url": {"url": _image_data_url(f)}})
        elif _classify_text(f):
            parts.append({"type": "text",
                          "text": _inline_text(f) or _text_stub(f)})
        else:
            parts.append({"type": "text", "text": _binary_stub(f)})
    return parts or ""   # empty submission -> "" for the blank-prompt guard
```

**Why not push the seam down into `app/openrouter.py` instead:**
- That would make the *engine* aware of file formats, which is the wrong direction — the engine's job is "given a `messages` list, POST it." Pushing file-shape logic into the engine couples it to `streamlit`'s `UploadedFile` type and forces the engine to know what a PDF is.
- The helper layer is *already* the place where prompt-shape decisions live (`_build_messages`, `_truncate_history`, the two-profile system prompt). The widening sits naturally next to those.

**The order of operations (mirrors the 6-priority roadmap in row 15):**
1. **P1 — friendly stub + 64 KB cap.** `web/chat_helpers.py` only. The model starts saying *"this file is a PDF, the helper cannot read it yet — please paste the relevant text"* instead of *"I cannot read PDF files."* Zero engine change, zero test regression. ~30 min.
2. **P2 — extract text from PDFs / DOCX / XLSX / PPTX.** New private `_extract_text(file) -> str | None` in the helper, called by `_format_upload_block`'s text branch. No new public signature. Adds 2-3 tests. ~2 h including a `pypdf` install.
3. **P3 — the schema widening.** The actual return-type change. ~6 new tests (one per part type, plus an empty-input regression, plus a "list passes through `_build_messages`" contract test). ~4 h including the engine `_build_payload` branch.
4. **P4 — vision model in `FREE_MODEL_CHOICES` + `supports: "vision"` flag on router slots.** Engine + view. ~2 h.
5. **P5 — `OPENROUTER_ALLOW_NON_FREE=1` opt-in.** Env-var gate in `app/router.py` + one sidebar checkbox. ~1 h.
6. **P6 — file persistence across turns.** `~/.stage1_uploads/<session_id>/<sha256>.<ext>` directory; the helper memoizes by sha256; the view exposes a "files in this chat" panel. ~4 h.

**Lesson (the durable one):** *when a transport feature ships cleanly and the next user request is "but the model can't see the contents," the bug is not in the transport — it is in the content shape one layer up.* A "the model can't see it" symptom in a multimodal pipeline is almost always a missing return-type widening at the prompt-building seam, not a model failure and not a wiring failure. Find the seam, widen the return type, leave the rest of the architecture alone. The seam in this project is `build_user_turn_text`; that is what row 15's P3 widens.

**Verification (this decision is advisory; P1-P3 are the next three engineering sessions):**
- No code change in this turn. The current shipped behaviour is row 14's `build_user_turn_text -> str` with the 12 KB binary stub — adequate for text files, honest about the gap for binaries.
- Test suite: **90/90 green** (unchanged from row 14). The widening is forwards-compatible: the helper's existing return type is a strict subset of the planned `str | list[dict]`, so adding the `list[dict]` branch cannot break a caller that only handles `str`.
- `py_compile` clean on `app/`, `cli/`, `web/`, `tests/`. The seam symbol resolves at the same line (L347) before and after the planned widening.

### Decision 11 — Docs-lag-code discipline: phase docs ship in the same commit as the code (2026-06-15)

**Trigger.** This row exists because of a *meta* bug discovered while writing rows 16-18. The original row 15 framed Phase 11 as *"advisory mode, awaiting user direction — the 6-priority roadmap is the next three engineering sessions."* Within hours of that framing, the engineering happened: `app/file_processor.py` (342 lines), `build_user_turn_content` (the row-15-P3 seam widening), `select_model_for_request` (the row-15-P4 auto-upgrade), `_DEFAULT_FREE_VISION_MODEL = "nvidia/nemotron-nano-12b-v2-vl:free"` (the only free-tier vision model that worked in practice), 28 new `tests/test_files.py` tests, 15 new `tests/test_smoke.py` tests, all green. The doc that said "advisory" was now 4-6 hours out of date. A reader joining the project at the end of Phase 11 would have had to read `app/file_processor.py` to learn that P1-P5 were actually done — which is exactly the failure mode the per-phase narrative in §3 / §4 / §7 is supposed to prevent.

**Chosen:** the per-phase narrative (`docs/technical_write_up.md`), the cross-cutting change-log (`improvements.md`), the user-facing readme (`README.md`), and the personal learning log (`docs/my_first_ai_journey.md`) are **part of the same commit as the code** that ships the phase. Not the next commit, not a "doc fixup" commit two days later, not "I'll add the section when I get a chance" — the same commit. A subsequent doc pass is a *sign* the rule was broken, not a *substitute* for following it.

**Why one commit, not a "doc fixup" later:**
- A reader's mental model of a phase is built from all four documents at once. If the code is in `main` but the docs say "planned," the reader either (a) trusts the docs and re-implements work that is already done, (b) trusts the code and skips the docs, or (c) wastes an hour diff-ing the two. None of those is a good outcome.
- The editor's *"what was actually built"* view is a different cognitive frame from the developer's *"what I was about to build"* view. They only converge after a small pause for verification. The 4-6 hour lag between row 15's "advisory" framing and this row's "shipped" reality is the time it took for that pause to happen. The new rule forces the pause to happen *before* the commit lands, not after — i.e. the developer writes the row, sleeps on it, verifies against the actual diff, and *then* commits the code.
- The "doc fixup later" pattern silently grows. Each phase adds a 2-day doc lag; a project that ships 10 phases ends up with 20 days of accumulated doc drift. The cost is paid by the next reader, not by the lag-creator.

**The mechanism (a 4-step checklist for every phase):**
1. **Code is merged.** The diff that ships the feature is in `main`, all tests green, the changelog bump is in `requirements.txt` or wherever it lives.
2. **`improvements.md` is updated in the *same* commit.** Section B (or its successor) gets a new dated subsection that mirrors the code's structure. Section C (or its successor) gets the SHIPPED / not-SHIPPED markers flipped. The Quick Reference table gets the new rows.
3. **`docs/technical_write_up.md` is updated in the *same* commit.** New rows for the new problem(s) solved, new decisions for the new structural choices, §8 Phase checklist gets the new phase entry, the "Last updated" footer gets the new timestamp.
4. **`README.md` and `docs/my_first_ai_journey.md` are updated in the *same* commit.* The user-facing surface (README) and the personal reflection (journey log) are the last two files to be touched, but they are *part of the same commit* — not a follow-up.

**What counts as "the same commit" (the loose definition this project will use):**
- For solo development, a single commit with a `feat:` / `fix:` / `chore:` body that includes one or more of `improvements.md`, `docs/technical_write_up.md`, `README.md`, `docs/my_first_ai_journey.md` in the file list. A commit that touches only code, even if a doc-touching commit follows within the hour, is a rule violation.
- For pair / team development, the rule is the same: a PR is open with the code diff *and* the doc diffs in the same review request. A PR with the code diff only, even if a follow-up PR lands the doc diffs before review completes, is a rule violation.
- A "doc-only fixup" commit is allowed *only* if the previous phase's commit was already rule-compliant — i.e. the fixup is correcting a typo or a small inaccuracy, not back-filling a missed section. If the fixup is adding a new section, that's a sign the original commit was rule-violating, and the fixup should be rebased into the original commit's history (e.g. via `git rebase -i HEAD~3` and a `reword` / `edit` on the original commit's hash).

**The escape hatch (the only acceptable reason to lag):** the phase is *spike-shaped* — i.e. the developer is exploring an unknown and doesn't know what the code will look like until the end. In that case, the rule is *write the row in the spike commit with a `[SPIKE]` marker, replace the marker with a `[SHIPPED]` marker in the shipping commit*. The marker is the audit trail; a spike commit without a marker is indistinguishable from a rule violation.

**Lesson (the durable one):** *the four documents are the contract the project has with its future readers. A phase that ships without updating the contract is a phase that has been handed over with the rug still rolled up.* The 4-6 hour lag in this row is the upper bound; zero lag (docs in the same commit as the code) is the target. The 4-document checklist above is the *mechanism* for hitting the target, not just an aspiration.

### Decision 12 — Per-chat RAG: lazy FAISS, version-counter invalidation, score-threshold 0.30, scored-then-merge with a separate global corpus (2026-06-15)

**Trigger.** Phase 12 (rows 19-20) introduces the persistent chat history sidebar and the RAG layer. The structural question that fell out of the build: where does the vector index live, when does it build, how is it invalidated, what is "good enough" similarity, and what is the right *shape* for the trust boundary between uploaded files and the global security corpus. Each of those is a small decision; together they are a "shape" decision that future readers need to understand in one place rather than re-derive from `app/rag_store.py` and `app/rag_global.py`.

**Chosen:**

1. **One FAISS index per chat, not one global index for everything.** Each `RagStore(chat_id)` instance owns its own `faiss.IndexFlatIP`. A query to chat A never traverses chat B's index. **Cross-chat isolation is a first-class invariant**, pinned by `tests/test_rag.py:CrossChatIsolationTests` (3 tests). The cost (more indices, more memory) is the right trade for a personal-tutor chatbot: each chat is a study session, the data is small, and the isolation guarantee is what makes the trust story honest.

2. **Lazy index build.** The FAISS `IndexFlatIP` is *not* built at `RagStore.__init__`. It is built on the first `search()` call and cached on the instance. Rationale: most chats never get a file upload, so allocating a 384-dim index for every chat would be wasteful. The first-search cost (re-encoding all chunks) is one-shot and ~50 ms on the smoke corpus.

3. **Version-counter invalidation, not manual rebuild.** The `chats` table has a `_index_version` integer column. `add_chunks_returning_ids` increments it. `RagStore.search()` compares the cached counter to the live counter and rebuilds the index if they differ. No explicit `rebuild()` API is exposed. The version counter is the single source of truth for "is the cache stale?" — there is no second path to invalidation.

4. **Score threshold = 0.30 (cosine similarity, after L2 normalization).** Below 0.30, a chunk is treated as noise and dropped before the context-injection step. Chosen empirically against a 5-query smoke corpus: 0.30 is the floor above which a chunk is "more similar than not" to the query, and below which the chunk adds context the model has to *filter out* before answering. The threshold is a single constant (`DEFAULT_SCORE_THRESHOLD` in `app/rag_embedder.py`) so it is trivially tunable. The trade-off is recall vs. precision: 0.30 errs on the side of fewer-but-cleaner chunks.

5. **Degraded mode via `PUKU_RAG_OFFLINE=1`.** When the env var is set, the embedder returns a 384-dim zero vector and the store short-circuits `search()` to an identity search (most-recent chunks, score 0.0). The point is *operational*: the test suite runs on CI without a sentence-transformers model download, and a developer without a working model registry can still exercise the retrieval pipeline. The model registry is the only piece that fails in degraded mode — chunking, storage, score-threshold logic, source citations all still work.

6. **Two parallel indices (per-chat + global), not one merged index.** The global security corpus (`app/rag_global.py:GlobalCorpusStore`) is its own FAISS index, populated once by `scripts/ingest_security_corpus.py` and deduped by SHA-256. The two indices are queried *separately* and the results are merged **scored, not concatenated** in `web/chat_helpers.py:build_messages_with_rag`: per-chat top-k first, global top-k second, then the helper picks the top-N across both (default 3 from per-chat + 2 from global). The two-index shape is what keeps the trust boundary explicit: a query about the user's PDF does not retrieve the global corpus's OWASP entry, and a query about OWASP does not retrieve the user's PDF.

7. **Context injection as a synthetic prior `user` turn, not a system prompt.** The helper prepends a `user` turn with text `"Context: <chunk 1>\n---\n<chunk 2>\n..."` to the messages list. The model sees the retrieved context as a *prior turn* in the conversation, not as instructions. This is intentional: it avoids fighting the model's instruction tuning, and it keeps the model-side prompt simple (one synthetic turn, one real turn, one assistant turn).

**Why not a single merged index:** a single index would conflate two trust boundaries (user uploads vs. reference material) and would also conflate two retrieval-quality concerns (per-chat chunks are dense in *one topic* — the user's PDF; global corpus chunks are sparse across *many* topics). Two indices, scored-merged, lets each index be tuned for its own distribution and lets the view surface both with distinct `source_kind` labels.

**Why not a system-prompt context block:** the model is instruction-tuned, and a long system-prompt block competes with the actual instructions ("be a friendly tutor", "use the teaching mode"). A prior-user-turn framing sidesteps the competition and is robust across model swaps.

**Why lazy + version-counter instead of always-rebuild:** the cumulative cost of rebuilding on every `add()` for a chat that gets 50 file uploads is ~50 * 50 ms = 2.5 s, all spent synchronously before the first search. The version-counter pattern defers the rebuild to the first search *after* the cache goes stale, and amortizes the cost over the user's next query. It also makes `add()` O(1) from the caller's perspective — no surprise latency.

**Lesson (the durable one):** *the shape of the RAG layer is the shape of the trust story.* Per-chat isolation says "your files are yours." Two indices says "the corpus is reference material, not your data." Scored-then-merge says "the model saw both, with provenance." Version-counter invalidation says "the cache is never silently stale." Each of these is a small decision; together they make the assistant auditable.

- `tests/test_rag_global.py` (23 tests, all green) — `GlobalIndexConstruction`, `AddAndSearch`, `SourceFilter`, `ClearAndReAdd`, `DegradedMode`, `CrossSourceMerge`, `EdgeCase`.

### Row 17 — Phase 12 / Session 8: copy-to-clipboard bubble buttons (2026-06-15)

**Trigger.** The user reported two visual bugs in the running web UI after Phase 11 shipped: (1) a JavaScript snippet was leaking visibly after every assistant reply, and (2) the copy-to-clipboard button on each bubble did not work. Both bugs had the same root cause: the original copy-button helper emitted an inline `onclick="var text=&quot;PAYLOAD&quot;…"` attribute, and the HTML parser terminated the attribute at the inner `"`, leaking the rest of the handler as visible text inside the bubble. The copy handler itself was still functional (the leaked snippet was a working handler), which is what made the bug subtle.

**Fix (structural, not cosmetic):**
- **Never put user-derived payload in an HTML attribute that needs to round-trip through `unsafe_allow_html`.** Use `data-text="<html-escaped>"` and let the browser's `dataset.text` getter reverse the escaping at the JS boundary.
- **HTML-escape the attribute value once on the Python side** with `html.escape(value, quote=True)` — this escapes `& < > " '` (Python 3.13).
- **Register the click listener exactly once, on `document`**, with `event.target.closest('.bubble-copy-btn')` matching. No per-button `addEventListener`.
- **Make the init script idempotent on both sides.** Python: module-level `_COPY_BUTTON_INIT_EMITTED` flag short-circuits the second-and-later call to `""`. JS: `window.__secMentorCopyBtnWired` flag stops re-registration if the script body somehow leaks in twice.
- **Per-element debounce.** `btn.__copyBtnBusy` flag prevents double-click storms from firing two copies.

**Code locations (the two-helper split):**
- `web/chat_helpers.py` — `_copy_button_html(text)` (pure, no JS in any attribute) and `_copy_button_init_script()` (the one-time `<script>` block with the delegated listener).
- `web/streamlit_app.py` — line 631 renders the init script after the CSS render; line 1246 wires `_copy_button_html(content)` into the assistant bubble render; `.bubble-copy-btn` CSS lives at the bottom of `_CUSTOM_CSS` (~line 478).
- `tests/test_smoke.py` — `CopyButtonHtmlTests` (10 tests) and `CopyButtonInitScriptTests` (10 tests) pin the new contract.

**Design doc:** the new `docs/copy_button.md` captures the full architecture, the failure mode that triggered the rewrite, the two-helper split, the XSS analysis, the idempotency story, and the test contract.
